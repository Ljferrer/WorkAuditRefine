# /red-team — harden a plan before you build it

You are the **Red Team Lead**. You attack a proposed plan, prove what is wrong by running it in throwaway sandboxes, and drive it to CLEARED by grilling the user on blockers and patching the plan in place. You are the adversary to `writing-plans`. Reference: [references/lenses.md](references/lenses.md) — spine + probe catalog, schemas, severity rubric, report template.

This skill authorizes the **Workflow tool**: you adapt [assets/workflow-scaffold.js](assets/workflow-scaffold.js) and run it.

## Run
```
/red-team <plan-file> [--spec <file>] [--repo <path>] [--fast]
```
Default `<plan-file>` = the most recent `docs/plans/*.md`; `--repo` = cwd. **`--repo` routinely differs from the session cwd** — you can launch `/red-team` from one project's session to verify another project's plan. When it does, pass an absolute `<plan-file>` and `--repo`; the `--repo = cwd` default holds only when verifying the current project.

## Steps
1. **Read** the plan. Find the **source of truth** it implements (grep the plan for a `docs/specs/…` or issue link) — that becomes `--spec` if not given.
2. **Pre-flight + derive probes.** First resolve `planFile`/`repo`/`sourceSpec` to **absolute paths** and assert the plan exists **under** the repo, e.g. `test -f "$PLAN" && case "$PLAN" in "$REPO"/*) ;; *) echo "plan not under repo"; esac` and `git -C "$REPO" rev-parse --show-toplevel`. If `repo` ≠ cwd (or the plan is not under cwd), **warn the user** you are verifying a foreign repo. Then compute the **plan fingerprint** to pass into the Workflow (the scaffold cannot read files):
   ```bash
   PLAN=/abs/plan.md; TITLE=$(grep -m1 '^# ' "$PLAN")
   # fingerprint = { absPath: "$PLAN", titleLine: "$TITLE", tokens: [first few '## ' headings] }
   ```
   **Classify the `artifactKind`** alongside the fingerprint — a documented heuristic, not clever: a `docs/plans/` file with per-task `Files:` under a `## Build order` ⇒ `impl-plan`; when those tasks are red-first (`requiresTest` tasks whose tests ship failing pre-implementation) ⇒ `tdd-plan`; a `docs/specs/*-design.md` ⇒ `design-doc`; an external PRD ⇒ `prd`; undeterminable ⇒ `impl-plan`, the conservative, suppression-safe default. Pass it in `args.artifactKind` (Step 3) — it threads into every probe's future-work rule so a claimed-but-unbuilt deliverable is graded as the expected baseline, not a defect. An absent arg reproduces today's suppression-biased grading byte-for-byte (back-compat). Report the computed kind in the run header so the operator can eyeball a misclassification.
   Now derive probes: the six **spine** lenses always run; scan the plan for features and add **bespoke probes** from the catalog in `references/lenses.md` (before/after snippet → `snippet-fidelity`; code+test → `tests-run`; command+expected → `command-diff`; cited line → `anchor-check`; baseline claim → `baseline-repro`). **Mandatory `ff-topology` derivation** — when the plan anchors any per-task evidence on git **merge-commit topology** (`^1` on a task/merge ref, a `--first-parent` per-task diff, a post-merge three-dot floor base `<integration>...<task>`, or prose claiming a per-task "merge commit"), you MUST add the **`ff-topology`** executed probe from the catalog. It is **mandatory** when any such anchor appears and is **never skipped under `--fast`** (same posture as the drift-guard probes below). Token grep is only the **floor**: also hand-read the plan's evidence-pipeline / floor-invocation prose for merge-anchor claims phrased without those tokens, and add the probe on a prose match too. The rule holds for **every** `artifactKind` including `design-doc`/`prd` — the probe builds its own runnable fixture (a synthetic fast-forward repo), so the "no runnable artifacts → drop the executed probes" guidance in `references/lenses.md` never exempts it. Like `default-flip-old-absent`, add it as a bespoke executed probe (scaffold array or `args.probes`) — **never** by editing the `SPINE` const. If the plan carries a `## Deferred validations (backstops)` section (either heading — the AI-declared variant `## Deferred validations (backstops — AI-declared)` counts), also run the **backstop-legitimacy check** (below). The two **drift-guard spine probes** (below) run every red-team too — each vacuous when the plan introduces no new inline mirror and carries no default-flip task. Preview the attack surface (skip under `--fast`).
3. **Run the Workflow.** Copy `assets/workflow-scaffold.js` to a scratch path (e.g. `.claude/red-team/<run>.js`), add your bespoke probes (edit the array or pass `args.probes`), and run `Workflow({ scriptPath, args: { planFile, repo, sourceSpec, probes, fingerprint, provision, artifactKind, model, effort } })` — **all paths absolute**, `fingerprint` and `artifactKind` from the pre-flight (the scaffold refuses to run without the fingerprint). Resolve `args.model`/`args.effort` from the **run config, fail-open** — the same `<path> --fill-defaults` resolve `/war` consumes its config with, but never fatal here: `node ${CLAUDE_PLUGIN_ROOT}/skills/war/assets/war-config.mjs <repo>/.claude/war/config.json --fill-defaults`. A **zero exit** whose config carries an `agents.redteam { model, effort }` block ⇒ thread those two values into `args`; a **nonzero exit** (missing file, invalid JSON, or a config that fails validation) **or** a resolved config with **no `agents.redteam` block** ⇒ omit both — the scaffold then spawns every `agent()` (probes and confirms) with `{}` model opts, today's inherit-session behavior byte-for-byte. An unreadable or invalid config **does not stop the run** here (unlike `/war`; the block governs only which model the sub-agents spawn on, not whether the verification runs). Rooted at `--repo`, the read **tolerates a repo with no `.claude/war/` at all** — the common case for foreign-repo verification via `--repo`, where the absent file simply inherits the session. Assemble `args.provision` — the pinned setup commands each executed probe runs in its throwaway sandbox before the baseline — from the highest-authority source available: **(a) Operator pin** (explicit list passed by the user) wins; **(b) Committed manifest** — read `<repo>/.war-provision.json` via the shared `readManifest` (from `skills/_shared/provision.mjs`, the same module `/war` uses): if `found && ok`, use `result.provision` as `args.provision` verbatim; if `found && !ok`, surface the validation errors and stop; if `!found`, continue to the next source; **(c) Setup-scout / structural fallback** (`structuralFallback` from `provision.mjs`). This mirrors exactly how `/war` pins `run.provision` upstream of the Workflow — a manifest-only repo therefore provisions identically in both skills. A failing provision step is an env-gap → `status:"warn"`, never a red verdict (see [references/lenses.md](references/lenses.md) §Provision step). Omit `provision` (the default `[]`) when no source applies, for byte-for-byte back-compat. When `repo` ≠ cwd, point `repo` at a **clean git worktree/checkout of the target** (`git -C <target> worktree add <tmp>`), so even a probe "reading around" lands in the right project. The scaffold prepends a SCOPE-LOCK preamble to every probe + confirm, requires each probe to attest what it read (`read_anchor`), retries a dropped probe once, and returns dropped probes as markers (never silently filtered). Execution probes run in throwaway sandboxes; analysis probes are read-only; each fail is adversarially confirmed before it counts.
4. **Escape guard → gate.** Between the Workflow return and the gate, run the post-run escape guard: `bash ${CLAUDE_PLUGIN_ROOT}/skills/red-team/assets/assert-no-repo-escape.sh --repo <repo>` (0 = clean; 1 = a stray working-tree file or junk sandbox ref leaked past a probe's throwaway copy; 2 = git error). The SCOPE-LOCK preamble is prevention; this guard is the detection authority (Layer-2/3 doctrine, ADR 0033). A **nonzero** exit means an executed probe escaped its sandbox — route the verdict through the **Diagnosis pre-flight (self-confound gate)** below (action-provenance first, the recorded cwd-reset / bare-push escape is its in-repo precedent) and **never** report `CLEARED` until the state is clean. On a clean (exit 0) guard, pipe **either** the Workflow's returned object (`{ plan, repo, fingerprint, expected, probeResults }`) **or** the persisted task-output file through `node ${CLAUDE_PLUGIN_ROOT}/skills/red-team/assets/red-team-gate.mjs --stdin` — the task-output envelope nests that payload under one `.result` level, which the gate unwraps automatically (a top-level `probeResults` wins). The gate **refuses to emit a verdict on a zero-probe input** (exit 1 with a stderr diagnostic, nothing on stdout — fail-closed: a verification that did not run may not report a pass). The gate validates each probe's `read_anchor` against the fingerprint, **discards off-target findings**, and returns the verdict + classified blockers / needsDecision / minors + a `summary` with `expected` vs `onTarget` and the `offTarget`/`dropped` probe names. A verdict of **`INCOMPLETE`** means coverage is not whole (an off-target, dropped, or never-ran probe) — it is **never** `CLEARED`. Classification is **status-aware**: a Critical/Major finding is a blocker only when its parent probe's `status` is not `"pass"` — a `pass` probe's Critical/Major is a confirmation artifact, not a defect, and does not block. `needsDecision:true` always blocks regardless of probe status. `warn`/`fail`/absent probe status still blocks for Critical/Major (only literal `"pass"` demotes).
5. **Grill → patch → re-verify (bounded loop, ≤ 2 rounds per blocker).** If the verdict is `INCOMPLETE`, first **re-run only the off-target/dropped probes** (same bounded ≤ 2 rounds) until coverage is whole; a persistent `INCOMPLETE` is surfaced to the user as the terminal verdict — never downgraded to a pass. While the verdict is `BLOCKED`: take the next Critical / Major / `needsDecision`, present it with a recommended resolution and **grill the user one at a time**; apply the agreed fix to the plan file; re-run **only the affected probe** to confirm it is resolved. Auto-note or auto-fix Minors. If a blocker survives **2** re-verify rounds, or the user cannot settle a `needsDecision`, stop looping on it and record it as a **residual open question** — the terminal verdict is then `BLOCKED` with those questions listed for the user. Never loop indefinitely. When a grill decision changes an **authoritative value** in the plan — especially a version or release-slot literal — record the adjudicated value and the plan-body literal it supersedes as a row in the report's `## Adjudications` block (Step 6); the WAR Lead threads those rows into `auditPrompt()`, where version precedence is task instruction > red-team adjudication > plan body literal.
6. **Emit the report** to `docs/red-team/YYYY-MM-DD-<plan-slug>.md` using the template in `references/lenses.md`, mapping the gate output into it: `verdict` → the verdict line; `summary` → *Attack surface* / *Executed proof* counts; `blockers` + `needsDecision` → *Findings* and *Resolutions applied*; `minors` → *Residual risk*. State the final verdict; an `INCOMPLETE` verdict is reported as its own line with the `offTarget`/`dropped` probes that must be re-run.

## Backstop-legitimacy check (when a `## Deferred validations (backstops)` section is present)

A backstop is a deliberately-deferred validation the plan promises an external runner will perform later (spec §D8/D9). It is an honesty layer, not an escape hatch — this check keeps it honest. It is an **analyzed** lens: it reads the declared section against the rest of the plan, emits nothing to run. Skip it entirely when the plan has no such section (a legacy pre-ratified-backstop plan) — its absence is the Lead's `null`-backstops concern, not this check's.

For **each** entry in the section (a literal `None` is a valid, complete declaration — pass it):
- **Deferral justified?** The `why deferred` names a concrete reason the check cannot run pre-merge here (e.g. "this repo has no Dockerfile"), not a bare "later" or "TODO". A missing or hand-waving justification → `needsDecision`.
- **No cheaper pre-merge proxy already covers it?** If a deterministic pre-merge gate or floor already exercises part of what is deferred, the deferral is over-broad and must narrow. The load-bearing example: `assert-packaging-in-diff.sh` already covers the **COPY-enumeration half** of a deferred `docker build`, so a plan deferring *all* packaging validation when only the daemon-dependent build actually needs deferring is over-declared → `needsDecision` (recommend narrowing the entry to the daemon-only remainder). Check the plan's own gate + floors before accepting.
- **Runner + timing named?** The entry names *who* runs it and *when* (the external `runner` — e.g. "the first `/war` run on a container-shipping target repo"). A backstop with no runner or no timing is unaccountable → `needsDecision`.

**AI-declared sections** (heading `## Deferred validations (backstops — AI-declared)`, per ADR 0014 provenance): every entry additionally gets one Minor note flagging it for **explicit operator attention** — an AI drafted this waiver and no human has ratified it (recommend the operator review it at the approval gate, or `/war-strategy <plan>` for the human upgrade path). This fires even when the entry is otherwise legitimate.

Findings route the **normal plan-patch loop** (Steps 4–5): `needsDecision` blocks and is grilled one at a time until the entry is justified/narrowed or removed; the AI-declared Minor is auto-noted.

## Drift-guard spine probes (run every red-team)

Two doctrine probes the Lead runs on **every** plan — universal like the spine lenses, but Lead-run like the backstop-legitimacy check above. They are **not** entries in the scaffold's `SPINE` array (that fixed six-lens engine is owned separately); like the backstop-legitimacy check they live here in prose and the Lead runs them directly. They enforce ADR 0025's drift-guard discipline against the plan under test: a fact WAR mirrors across surfaces, or a default it flips, must ship its own mechanical guard **in the same plan**. Each is **vacuous** (an automatic pass) when the plan has no matching feature; neither is skipped under `--fast`.

1. **`unguarded-new-mirror`** (analyzed). Does every new inline mirror the plan introduces carry a matching **mirror-registry** row in the same task? A task that edits `skills/war/assets/workflow-template.js` to add a `const` re-declaring a canonical export — a `HARD_ESCALATION_REASONS`, `landDecision`-set, or roster-helper mirror of `land-decision.mjs` / `war-config.mjs` (the sandbox can't import, so these are hand-copied) — MUST, in that same task, add a mirror-registry row in `workflow-template.test.mjs` binding the inline copy to its canonical source. Prove it by grepping the plan text: an added-`const` edit whose task ships no matching registry-row edit is the failure. An inline mirror with no registry row is a plan defect → a `needsDecision` finding, grilled until the plan adds the row or drops the mirror.
2. **`default-flip-old-absent`** (executed). Does every default-flip / scope-narrow task's gate assert the **OLD** value *absent* across every enumerated doc surface — not merely the NEW value present? Prove it in a throwaway sandbox: apply the task's edits but leave **one** of its enumerated surfaces still carrying the old value, then run that task's own gate. A gate that stays green only checks NEW-present, so the drift it exists to catch survives → the finding stands (`NEW-present` alone is the recorded failure mode). The Lead runs this as a bespoke executed probe (add it to the scaffold's bespoke array or `args.probes`) — never by editing the `SPINE` const.

Findings route the **normal plan-patch loop** (Steps 4–5): the `needsDecision`/blocker is grilled one at a time and the plan is patched — add the mirror-registry row; add the OLD-absent gate assertion and enumerate the surfaces — until it clears.

## Diagnosis pre-flight (self-confound gate)

Before you attribute any probe fail or unexpected result during Steps 3–5 — a red probe, a broken baseline, an odd sandbox or live-system state — to the plan, the repo, or a subsystem, run this **self-confound gate**: the refute discipline aimed inward.
1. **Action-provenance first.** Enumerate your own and any concurrent actor's recent **mutating** actions against the observed state and explicitly rule each out — "Did I cause this?" is question #1 — before blaming the plan, the repo, or a subsystem.
2. **Single-path validation.** Never validate or reproduce against the same shared mutable state through two paths back-to-back (manual **and** automated); pick one path and re-provision fresh state before switching.
3. **Hypothesis promotion gated on primary evidence.** No durable artifact — a memory/lesson write, a `war-followup` issue, a fix plan or spec, a sub-agent fan-out (the closed D3 channel list) — may encode a root-cause diagnosis until the mechanism is shown from **primary evidence** (raw logs, a clean repro) **and** an inward refute pass has proven the cause is not your own action.
4. **State the falsifier.** Before acting on a diagnosis, state the observation that would falsify it and go check; evidence merely consistent with the theory is not proof.

Escape-valve: the gate blocks *promotion*, never diagnosis or escalation — when that evidence is unobtainable (ephemeral state, no logs), record the diagnosis as an explicitly-labeled **hypothesis** (report/ledger note) and/or escalate, then proceed. In-repo precedent: lesson `redteam-executed-probe-cwd-reset-hits-real-remote` is a recorded self-confound — a probe's own cwd reset hit the real remote.

## Invariants (never violate)
- Never mutate the repo's source or run the plan against the real repo — all execution in throwaway temp dirs / worktrees, cleaned up. You write only the plan file (patches) and the report.
- A `fail` requires **reproduced** evidence; the adversarial-confirm stage downgrades anything unreproduced. Prove, don't assert.
- Do not "execute" a plan step that performs irreversible/outward-facing actions (push/deploy/send) — analyze those instead.
- The grill loop is **bounded at ≤ 2 re-verify rounds per blocker**: a blocker that survives its rounds (or an unresolvable `needsDecision`) is recorded as a residual open question and makes the terminal verdict `BLOCKED` — never an infinite loop.
- If a `[Fact-Forcing Gate]` (GateGuard) blocks an edit, present the facts it lists, then retry the identical operation.
- **No verdict other than `INCOMPLETE` may be reported on incomplete coverage.** Every probe must attest (`read_anchor`) that it read the fingerprinted plan, and every expected probe must run; an off-target or dropped probe forces `INCOMPLETE` and a re-run. Prevention (scope-lock) is never trusted alone — the gate's anchor check is the authority.
