# Partial-phase recovery & task-branch hygiene — adopt held work, re-dispatch one task, and make every ref proof fail loud

**Issues addressed:** #725 (whole), #728, #731 (residual only), plus the **engine half of #650** (stale remote task branches on restart; the red-team ff-topology half of #650 is out of scope here — see §9).

**Survey group:** Partial-phase recovery & task-branch hygiene (Survey Corps, 2026-07-11).

## 1. Context — the gap / problem

A `held:escalation` partial phase (observed live: campaign `2026-07-08-memory-frictions`, plan 9, run `wf_20ff5e9e-6a4` — 6 of 7 tasks merged to `integration/<slug>/phase-1`, one task escalated, gate-audit HARD-held, `landResult` null) strands real work behind four engine gaps, and two adjacent ref-hygiene defects make recovery attempts riskier than they need to be:

1. **Non-empty orphaned integration branch blocks relaunch** (#725.1). `cmd_ensure_integration`'s `--reclaim-empty-orphan` (in `skills/war/assets/provision-worktrees.sh`) by design only deletes *proven-empty, origin-absent* orphans. A partial-phase integration branch carrying 6 real commits hits the `EX_FOREIGN` die, and the only recovery is hand-editing the `--owned-file` ledger or deleting the branch — the die message even names a `record-as-owned` verb that does not exist as a subcommand.
2. **No first-class single-task re-dispatch** (#725.2). Resume today is whole-run; the manual alternative is the Lead-run escalation-completion-land recipe in `skills/war/SKILL.md`. There is no engine path that re-dispatches *only* the escalated task against the integration tip and re-runs the phase-close gate-audit.
3. **Plan-defect escalations are indistinguishable from implementation-defect escalations** (#725.3). The observed escalation (`fixRounds: 0`) was a plan/spec flaw a worker cannot fix; WAR surfaced it as a generic escalation with no signal routing it to `/red-team` / plan amendment instead of worker fix-rounds.
4. **Missing local working-branch ref dies cryptically at the provision barrier** (#725.4). When fetch succeeds but the local base ref is absent (origin-only branch), `cmd_ensure_integration`'s reconcile block is skipped (`[ -n "$origin_sha" ] && [ -n "$local_sha" ]` is false), `cut_ref` stays the bare name, and `git branch` dies "not a valid object name" — with no origin fallback and no hint (lesson `war-provision-barrier-needs-local-working-branch-ref`).
5. **Stale remote task branches block restart pushes** (#650 engine half). Tearing down *local* task branches + worktrees between reruns leaves the *remote* `war/<slug>/pN-tK` at the prior run's output. The relaunch worker's push is rejected non-fast-forward; the worker correctly refuses to reconcile (the `FORCE_WITH_LEASE_RULE` carve-out covers only dispatch-rebase divergence) and escalates — correct behavior, but it costs a full restart to diagnose. `cmd_ensure_worktree` cuts/reuses only local branches and never probes origin; `remove_worktree` never touches branch refs; nothing deletes remote task branches.
6. **The reclaim safety proof reads a git error as "proven empty"** (#728). In `cmd_ensure_integration`'s reclaim block, `orphan_commits="$(git log --oneline "$base..$branch" 2>/dev/null || true)"` swallows any `git log` failure into empty output. An unresolvable `<base>` falsely passes proof 1, and combined with proof 2's origin-absent reading, `git branch -D` can delete a branch that carries unique commits — undercutting the exact ADR 0008 invariant the two-proof discipline exists to guarantee. (Also recorded as lesson `reclaim-empty-orphan-proof-swallows-git-log-error-as-empty`.)
7. **Manual lands get no follower-sync assertion** (#731 residual). The automated land path already mechanizes the 6-recurrence lesson `land-local-follower-ref-can-lag-sync-before-next-phase`: `cmd_land_advance` performs a post-push origin readback and a follower CAS `update-ref` (including the ALREADY-LANDED reconcile branch), and `cmd_ensure_integration`'s create path reconciles local-vs-origin before cutting (landed in #593). **Cite that mechanization; do not rebuild it.** The residual gap is the *manual* escalation-completion-land path: if the Lead deviates from the recipe (a raw `git push` instead of `land-advance`), nothing asserts the local follower matches origin afterward.

## 2. Pivotal constraints

- **ADR 0008 — never destroy work; git > labels > ledger.** Recovery adopts and repairs *records toward git*, never git toward records; an unexplained commit halts. No auto-delete of anything carrying unique commits; every destructive step needs its proofs, and every proof must **fail loud on command error** (a git error is never "empty" / "absent").
- **ADR 0005 — never add `held:workflow-error` to `HARD_ESCALATION_REASONS`; enum discipline.** No new members of `HARD_ESCALATION_REASONS` or `KNOWN_LAND_DECISIONS` in this spec at all. Any touched enum surface changes the canonical export in `land-decision.mjs`, its hand-mirrored copy inside `workflow-template.js`, and the drift-guard test **together** (`land-decision.test.mjs` is the arbiter).
- **The no-force-push invariant.** Nothing in this spec force-pushes any ref. The one existing carve-out — `FORCE_WITH_LEASE_RULE` (canonical string in `workflow-template.js`, mirrored verbatim in `agents/war-worker.md`, byte-compared by test) — stays byte-identical and is not widened. Stale-remote reconciliation uses `git push origin --delete` + a plain `git push`, exactly the recorded Lead reconciliation in #650.
- **ADR 0003 — foreign/unowned refs fail loud by default.** All new reclaim/adopt behavior is opt-in, flag-gated, and Lead-supplied only on a *sanctioned recovery relaunch*, mirroring `--reclaim-empty-orphan`'s posture.
- **Prompt-surface split.** Any change to worker push-handoff behavior lands on **both** surfaces — `agents/war-worker.md` (standing) and the string-built prompt in `workflow-template.js` (dispatched) — in the same commit, with a mirror/token test.
- **Task status enum discipline.** `merged` is terminal and task-level; `landed` is phase-level only. The re-dispatch path classifies already-integrated tasks as `merged`, never invents a new status.
- **ADR 0023 — land asserts git ground truth.** The manual-land assertion reads origin via `ls-remote` (rc-checked — see lesson `ensure-origin-swallows-stderr-unlike-sibling-subcommands`), never the local follower, as truth.

## 4. Mechanics

### 4.1 `record-as-owned` — tooled adoption of a non-empty orphan (`provision-worktrees.sh`)

New subcommand: `record-as-owned <branch> <base> --owned-file PATH`.

- **Proof 1 (mechanical, lineage):** `git rev-parse --verify --quiet` both `<branch>` and `<base>` (die `EX_FOREIGN` on either failure — same fail-loud discipline as §4.5), then require `git merge-base --is-ancestor <base> <branch>`; refuse (`EX_FOREIGN`) when the orphan does not strictly descend from the supplied frozen phase base — a branch rooted elsewhere is foreign work, not this run's partial phase.
- **Proof 2 (procedural, sanction):** print the `git log --oneline <base>..<branch>` ahead-commits to stdout before recording, so the invocation is an *informed* sanction. The Lead's standing duty (SKILL.md runbook, §4.8) is to map every listed commit to a `merged` task in the run's audit log/ledger before invoking — an unexplained commit halts the recovery (ADR 0008 resume doctrine).
- On both proofs: append `<branch>` to the `--owned-file` ledger (reusing `record_owned_file`), idempotently (already-recorded → no-op, exit 0). **No ref is created, moved, or deleted** — this subcommand repairs the record toward git, nothing else.
- The existing `EX_FOREIGN` die messages that already direct the operator to record the branch as owned become true: `cmd_teardown_task`'s die names the literal `(record-as-owned)` verb, while `cmd_ensure_integration` and `cmd_teardown_phase` say "record it as owned" / "record the branch as owned" without the parenthetical — align all three on the literal verb name in this change. After adoption, relaunch takes the unchanged owned-reuse path ("Legitimate resume: reuse as-is. NEVER re-cut or move it.").

### 4.2 Single-task re-dispatch via git-derived merged-set (`workflow-template.js`)

On a **sanctioned recovery relaunch** (the Lead adopts the integration branch per §4.1 and relaunches the phase; the frozen phase base the Lead supplies is the *adopted integration tip*):

- **Pre-dispatch merged-set derivation:** before scheduling waves, for each task whose branch `war/<slug>/pN-tK` exists locally, test `git merge-base --is-ancestor <task-branch-tip> <integration-tip>`. On true, record the task `merged` (terminal task status — never `landed`, which is phase-level only) with a note `recovered: pre-merged on adopted integration branch`, and never spawn its worker. On false or absent branch, the task dispatches normally — its worktree is cut at the adopted integration tip (the relaunch's frozen phase base), which is exactly the "re-dispatch against the integration tip" the issue asks for.
- Derivation reads **git only** (ADR 0008: git > labels > ledger); issue labels and `ledger.json` are reconciled toward the git-derived answer, never the reverse.
- The rest of the phase is unchanged: the re-dispatched task gets the full Work+Audit loop (fresh worker, full auditor roster, pinned `audit_sha`, unanimous approval), the serial Refine merge, and the **post-merge gate-audit (`execution-evidence` lens) re-runs on the new integrated tip** — the recovery path earns the same evidence as a first run. Land proceeds through the unchanged push-first CAS.
- The merged-set derivation is skipped entirely on a first run (no task branches exist → derivation is vacuous), so non-recovery launches take an identical path (the §4.4 stale-remote classification clause is the barrier prompt's only always-on delta — it is default behavior, not recovery machinery).

### 4.3 Plan-defect escalation distinction (`workflow-template.js` + `agents/war-worker.md` + `land-decision` guards)

- **Sentinel:** the worker's "Stop and escalate instead of guessing" clause gains one sentence: when the block's root cause is a **plan/spec defect** (the plan contradicts the code, a specced construct cannot exist as described, an ambiguity with no intent-consistent resolution), prefix `blocked_reason` with the literal token `PLAN-DEFECT:`. Lands on **both** surfaces in the same commit — `agents/war-worker.md` (standing card) and the dispatched worker prompt string in `workflow-template.js` — with a both-surfaces token test in `workflow-template.test.mjs` (same idiom as the existing `FORCE_WITH_LEASE_RULE` mirror test).
- **Tagging:** where the Workflow pushes a worker-blocked escalation record, it sets `defectClass: 'plan'` when `blocked_reason` starts with the sentinel; the field is **absent** otherwise (never `'implementation'` by default — absence keeps prior-run records shape-compatible and avoids asserting a classification nobody made).
- **`defectClass` is metadata, orthogonal to `reason`** — exactly as finding `disposition` is orthogonal to severity. It never enters `decideLand`, `HARD_ESCALATION_REASONS`, or `KNOWN_LAND_DECISIONS`. It rides the escalation record into the machine-readable `handoff` block so the Lead (and `/aftermath`) can route it.
- **Negative drift-guard:** `land-decision.test.mjs` gains an assertion that neither `'plan-defect'` nor `'held:plan-defect'` is a member of `HARD_ESCALATION_REASONS` or `KNOWN_LAND_DECISIONS`, on **both** the canonical exports and the hand-mirrored inline copies in `workflow-template.js` — pinning ADR 0005 against the most likely future regression (someone "completing" this feature into the enum). `land-decision.mjs` itself changes only by comment: a note on the `HARD_ESCALATION_REASONS` block that defect classification is escalation-record metadata, never a reason.
- **Routing (prose, SKILL.md §4.8):** at `held:escalation` adjudication, `defectClass: 'plan'` (or a Lead reading the audit log and locating the defect in the plan slice) routes to **`/red-team` plan amendment** — never to worker fix-rounds and never to the escalation-completion land (there is nothing to "complete"; the plan is wrong). Vocabulary check performed: `plan-defect` collides with no war-memory provenance tier or existing enum/tag (grep across `skills/`, `agents/`, `docs/adr/` = zero hits at master).

### 4.4 Stale remote task-branch reconciliation (`provision-worktrees.sh` + `workflow-template.js` env-blocked mapping + both worker prompt surfaces)

- **Probe-classification (default):** in `cmd_ensure_worktree`, only on the **fresh-cut path** (local `<branch>` does not exist — the exact torn-down-locally-but-not-remotely restart shape), probe `git ls-remote origin "refs/heads/<branch>"` with the rc captured separately from output (a network failure **warns and proceeds** — offline runs must still provision; the worker's push-handoff escalation remains the backstop). A remote tip that **is** an ancestor of (or equal to) the frozen integration tip is already-integrated work — warn and proceed. A remote ref that is **not** an ancestor of the frozen tip is a *stale prior attempt* — a **per-task classification, not a barrier failure**. The script's own contract stays mechanically checkable: invoked directly on this shape, `ensure-worktree` **dies** (dedicated exit code, documented in the script's exit-code header block) with a diagnostic led by a stable machine-readable `STALE_REMOTE` marker line (remote SHA, frozen tip) and naming both recovery directions, mirroring the ensure-integration DIVERGED die: **(a)** adopt the remote as the real work — `git branch <branch> <remote-sha>` then relaunch (the existing-branch reuse path checks it out as-is); **(b)** on a sanctioned recovery relaunch where the remote is a superseded attempt, pass `--reclaim-stale-remote`.
- **Barrier routing (per-task soft outcome):** the existing provision-barrier refiner dispatch probes each task per the rules above and, instead of failing, reports classified per-task results on its env-outcome JSON — an optional `staleRemote: [{ task, remoteSha, frozenTip }]` list (the barrier keys on the `STALE_REMOTE` marker token, never the numeric exit code). The engine maps each entry to the **existing per-task `env-blocked` status** (worker never spawned) with the full two-direction diagnostic — adopt via `git branch <branch> <remoteSha>` + relaunch, or sanctioned `--reclaim-stale-remote` — plus the restore command. Siblings proceed; the phase ends in the normal held/handoff flow; recovery = adjudicate + relaunch with the merged-set skip (§4.2) re-dispatching only the blocked task(s); dependents of a blocked task follow the existing `dep-failed` semantics. **ADR 0021's all-or-nothing topology barrier is NOT restructured**: all cuttable worktrees are still cut from the one frozen base; the stale-remote result is env classification, the same family as `run.provision` failures.
- **Sanctioned reclaim (opt-in):** `--reclaim-stale-remote` on `ensure-worktree`, Lead-supplied only on a sanctioned recovery relaunch (threaded through the Provision steps the same way `--reclaim-empty-orphan` is). Proofs before acting: (1) the branch is in this run's exact `war/<slug>/pN-tK` namespace **by construction** (ensure-worktree only ever receives its own task branch names); (2) the local ref is **absent** (nothing local adopted the remote — if a local ref exists, the reuse path owns it and the flag is a no-op). On both proofs: `git push origin --delete "refs/heads/<branch>"` (rc-checked, die on failure), then cut fresh at the frozen tip as today. **No force-push exists on any path** — deletion + later plain push is the recorded #650 Lead reconciliation, now tooled. Without the flag, the classification above stands — per-task `env-blocked` at the barrier, the dedicated die on direct invocation (ADR 0003's fail-loud posture at the script surface).
- **Worker push-handoff clause (both surfaces, same commit):** one added sentence adjacent to `FORCE_WITH_LEASE_RULE` (which stays byte-identical): a non-fast-forward push rejection where the remote task branch was never merged and shares only an older base is a *stale prior attempt* — do **not** rebase onto it, merge it, or widen `--force-with-lease`; escalate with the remote tip SHA and the divergence base in `blocked_reason` so the Lead's reconciliation is one `git push origin --delete` away. Mirrored in `agents/war-worker.md` "Dep-wave rebase + force-with-lease carve-out" section and the dispatched prompt in `workflow-template.js`, with a both-surfaces token test.

### 4.5 Reclaim proof-1 hardening (#728, `provision-worktrees.sh`)

In `cmd_ensure_integration`'s `--reclaim-empty-orphan` block:

- Before the log proof: `git rev-parse --verify --quiet "$base^{commit}" >/dev/null || die … "$EX_FOREIGN"` — an unresolvable base is an operator error, never a pass.
- Replace `orphan_commits="$(git log --oneline "$base..$branch" 2>/dev/null || true)"` with an rc-captured form: run `git log` with its exit status captured separately from stdout; a non-zero rc dies `EX_FOREIGN` with git's own stderr surfaced (the `_tmp_err` idiom already used by the fetch path in this same function — see lesson `ensure-origin-swallows-stderr-unlike-sibling-subcommands`), and only a **rc-0 empty output** passes proof 1.
- Update the two-proof comment block above the code so it no longer describes the swallowed-error behavior (comment-lag rule: grep the touched function for `2>/dev/null` prose describing proof 1; **grep is a floor — also hand-scan the reclaim comment block and the test file's case 7a–7d titles/comments for descriptions of the old behavior, and list any stragglers as survey-derived corrections**).
- Regression test (`provision-worktrees.test.sh`, extending the reclaim case family): `--reclaim-empty-orphan` with an unresolvable `<base>` exits 3 and the orphan branch still exists with its tip SHA unchanged.

### 4.6 `sync-follower` — manual-land follower assertion (#731 residual, `provision-worktrees.sh` + `skills/war/SKILL.md`)

The automated path needs nothing: `cmd_land_advance` already performs the post-push origin readback, the follower CAS `update-ref`, and the ALREADY-LANDED follower reconcile; `cmd_ensure_integration`'s create path already reconciles local-vs-origin before cutting (equal/behind/ahead/diverged, landed in #593). This spec **cites that mechanization as the contract** and adds only the manual-path assertion:

New subcommand `sync-follower <branch>`:

- Read local: `git rev-parse --verify --quiet "refs/heads/<branch>"`. Read origin: `git ls-remote origin "refs/heads/<branch>"` with rc captured separately (rc ≠ 0 → die, network error is never "absent"; origin genuinely absent → die with a "nothing landed to origin — did the manual land push?" hint, which is precisely the deviation this assertion exists to catch).
- Equal → exit 0. Local strictly behind origin (`merge-base --is-ancestor local origin`) → guarded CAS fast-forward (`git update-ref` with expected old value), **skipped with a warning (still exit 0)** when the branch is checked out anywhere — byte-consistent with `cmd_ensure_integration`'s follower-ff discipline, and benign because the next phase's create path cuts from origin regardless. Local absent → create the follower at the origin tip (mirrors `land-advance`'s create branch). Local strictly ahead or diverged → die `EX_DIVERGED` with both SHAs and the two ADR 0008 repair directions (reuse the existing DIVERGED die wording). **Never force-pushes; never touches origin.**
- `skills/war/SKILL.md`'s **escalation-completion land** recipe gains one final step: `node …/provision-worktrees.sh sync-follower <working>` — an assertion that costs one command and catches a recipe deviation (raw `git push`) before the next phase consumes a stale follower. The `held:land-failed` auto-recover recipes (a) and (b) gain the same trailing step.

### 4.7 Missing local base ref at the provision barrier (#725.4, `provision-worktrees.sh`)

In `cmd_ensure_integration`'s create path, extend the post-fetch resolution: when `local_sha` is empty and `origin_sha` is non-empty, set `cut_ref="$origin_sha"` with a warn naming the origin fallback (no follower ff — there is no local ref to move; the follower stays absent, which `land-advance`'s create branch and `sync-follower` both handle). When both are empty, die with a diagnostic that names the missing-local-ref cause and the fix ("branch '<base>' has no local ref and could not be resolved on origin — create the local working branch or check the branch name; see lesson `war-provision-barrier-needs-local-working-branch-ref`") instead of the raw "not a valid object name" surfacing as `held:workflow-error`.

### 4.8 Recovery runbook (`skills/war/SKILL.md`)

One new subsection under the resume/held adjudication doctrine — the **held-partial-phase recovery runbook** — sequencing the tools above: (1) adjudicate the escalation (`defectClass: 'plan'` → `/red-team` plan amendment, stop here; implementation defect → continue); (2) map every `git log <base>..<integration>` commit to a merged task (unexplained commit → halt, ADR 0008); (3) `record-as-owned <integration-branch> <base> --owned-file <ledger>`; (4) relaunch the phase as a sanctioned recovery relaunch with the adopted integration tip as the frozen phase base, threading `--reclaim-stale-remote` so the escalated task's stale remote branch (if any) is reconciled at Provision; (5) the merged-set derivation skips the six merged tasks, re-dispatches the one, re-runs the gate-audit, and lands through the unchanged CAS path; (6) on any *manual* completion instead, finish with `sync-follower <working>`. The runbook cross-references the existing escalation-completion-land recipe as the "small, fully-understood Major" shortcut and positions the relaunch path as the default for anything larger.

## 6. New domain terms (CONTEXT.md)

- **Orphan adoption (`record-as-owned`)** — repairing the owned-file ledger toward git so a non-empty partial-phase integration branch becomes a legitimate resume target; the ADR 0008-conformant opposite of reclaim-deletion.
- **Stale prior attempt** — an unmerged remote `war/<slug>/pN-tK` cut from an older base, left by a prior run whose local state was torn down; blocks the identically-named relaunch push; surfaces at Provision as a per-task `env-blocked` classification (siblings proceed); reconciled by Lead-sanctioned remote deletion, never force-push.
- **Defect class** — escalation-record metadata (`defectClass: 'plan'`) distinguishing a plan/spec defect (routes to `/red-team` plan amendment) from an implementation defect (routes to fix-rounds / escalation-completion); orthogonal to escalation `reason`, never an enum member.
- **Sanctioned recovery relaunch** (extend the existing usage) — now also the context that licenses `record-as-owned`, `--reclaim-stale-remote`, and the merged-set single-task re-dispatch.

## 7. Recommended ADRs

- **ADR (new): Partial-phase recovery is adoption plus re-dispatch, never deletion plus re-run.** Records: the adopted integration tip becomes the recovery relaunch's frozen phase base; the merged-set is derived from git ancestry (never labels/ledger); destructive reconciliation (empty-orphan delete, stale-remote delete) is always flag-gated, two-proof, fail-loud-on-command-error; `defectClass` stays out of every land/escalation enum.
- No amendment to ADR 0005 or ADR 0008 is needed — this spec operates strictly inside both.

## 8. Open risks / implementation notes

- **Merged-set derivation vs. reused worktrees:** a recovered task branch that exists but is *not* an ancestor of the integration tip (e.g. the escalated task's half-done branch) takes the ensure-worktree existing-branch reuse path, so the re-dispatched worker sees prior commits — the resume doctrine (owned-file continuity, prior commits kept) already governs this; the plan must not add any reset.
- **`--reclaim-stale-remote` deletes unique commits by design** (the superseded attempt is unmerged). The sanction is the Lead-supplied flag on a recovery relaunch, and the safety floor is the proofs plus the default probe-classification — per-task `env-blocked` at the barrier, the dedicated fail-loud die on direct invocation — whose diagnostic forces the adopt-vs-discard adjudication first. The plan's tests must prove the flag is inert when a local ref exists.
- **Offline probe fail-open** (§4.4) means an offline relaunch can still hit the push rejection later; the worker's new escalation sentence is the deliberate backstop, not a gap.
- **`sync-follower` checked-out-behind warns and exits 0** — consistent with ensure-integration's follower-ff skip; if operators find the warn insufficient, tightening to a distinct exit code is a follow-up, not this spec.
- **Comment/prose sweeps:** every step that greps old-behavior terms (§4.5's `2>/dev/null` proof prose; any `record-as-owned` die-message wording; SKILL.md manual-land prose) carries the mandatory manual same-scope survey — hand-scan the touched functions' comment blocks, the shell test case titles, and the SKILL.md subsections for stragglers the grep misses, and list them as survey-derived corrections.
- Anchor all plan `Files:`/locators by **named construct** (`cmd_ensure_integration` reclaim block, `FORCE_WITH_LEASE_RULE`, "Stop and escalate" section), never line numbers — this file family sits in the serial merge queue.

## 9. Non-goals / deferred

- **#650 red-team half** (ff-topology spine probe / sandbox fixture) — belongs to the red-team file family (`skills/red-team/…`), a different survey group; not touched here.
- **Auditor-lens plan-defect sentinel** (`agents/war-auditor.md` + dispatched auditor prompt) — deferred; the Lead-adjudication prose (§4.3 routing) covers auditor-found plan defects until a red-team/auditor-family group picks it up.
- **Automated `/red-team` invocation on `defectClass: 'plan'`** — routing stays a Lead decision; no auto-invoke.
- **Restart-time remote cleanup as an unconditional default** — rejected, not deferred (destroys unadjudicated work).
- **Workflow-side `sync-follower` mention in the land dispatch** — the automated path already syncs via `land-advance`; adding prompt prose there is noise.
