# Launch entry validation completeness + engine-throw escalate routing

Source spec: docs/specs/2026-07-12-war-launch-entry-validation-design.md
Issues addressed: #740, #742

## Commander's Intent

- **Purpose:** Two live engine defects make errors surface as the wrong outcome deep in a run — a launch missing `ph.title`/`ph.workingBranch`/`ph.integrationBranch` dies opaquely inside pt-tagged prompt construction instead of at entry (#740), and an engine-side throw inside the work thunk is nulled by the live `parallel`, so the result is dropped, `nextWave()` silently re-dispatches a completed task every wave iteration (~660k tokens/round), and the task ends mislabeled `unrunnable-deps` (#742). Give both classes a named, boundary-time route.
- **Method:** Widen the Entry validation (H) block with an unconditional phase-field check feeding a hoisted `problems` aggregation whose `(or supply explicit branch/worktree per task)` suffix appends only when a derivation-class problem fired (route unchanged: throw → catch → `held:workflow-error`); enforce the wave-loop invariant — one dispatch, exactly one collected result — by wrapping the WHOLE per-task work+audit thunk body (including `provisionStep`) in try/catch returning `verdict: 'escalate'` (reusing the existing `'escalate'` HARD reason; ADR 0005 mirrors untouched), consciously converting the per-task provision evidence-gate no-evidence throw and inside-thunk pt throws from harness-only `held:workflow-error` to `held:escalation`; reshape `assertReportedPathsInWorktree` into `normalizeReportedPaths` (relative pass / this-worktree pass / sibling-worktree throw / `mainCheckout`-rooted rewrite+warn / other-absolute throw), reassigning `impl.files_changed` to the normalized array; and state the worktree-relative `files_changed` contract as one canonical `FILES_CHANGED_RULE` constant byte-mirrored across both prompt surfaces plus `schemas.md`, in one commit.
- **End state:**
  1. Launching with tasks carrying explicit `branch`/`worktree` but `ph.title` absent returns `held:workflow-error` whose message names `phase.title` and does NOT carry the `(or supply explicit branch/worktree per task)` suffix; zero agents spawned; git untouched. Same individually for `phase.workingBranch` and `phase.integrationBranch`; all three absent (or `ph` nullish) ⇒ one error naming all three.
  2. Trio absent AND `ph.workingBranch` absent (derivation-needing task) ⇒ exactly this single message: `workflow-template: requires top-level { planSlug, runId, worktreeRoot } — missing: [planSlug, runId, worktreeRoot]; workflow-template: requires phase { title, workingBranch, integrationBranch } — missing: [workingBranch] (or supply explicit branch/worktree per task)`.
  3. The existing `run-lifecycle §1 entry validation` cases (a)–(d) pass unmodified (every regex in (a)–(d), including (d)'s trio-class-NOT-reported distinctness assert, holds against the hoisted-aggregation refactor).
  4. Escalate, not re-dispatch: under a live-mirroring parallel (thunk rejection → `null` result), a mocked worker returning `files_changed` with an absolute path outside both roots and `mainCheckout` SET yields exactly ONE `work:<id>` dispatch; an `escalated` entry with `reason: 'escalate'` and the path-contract message verbatim in `blocked`; `landDecision: 'held:escalation'`; NO `unrunnable-deps` entry in `escalated` or `auditLog`.
  5. Normalization: a mocked worker reporting a `mainCheckout`-rooted path proceeds to audit; the WorkerResult object (the test holds the fixture reference) carries the worktree-relative form post-run via the `impl.files_changed` reassignment; a warning log line names the task and original path. `mainCheckout` unset ⇒ same report escalates per (4). An absolute path under `worktreeRoot` but under a SIBLING task's worktree ⇒ throws → escalates per (4), never normalizes.
  6. The per-task provision evidence-gate no-evidence result now escalates: the `run-lifecycle §3 evidence gate` no-evidence family is rewritten + retitled to assert `held:escalation` with the evidence-gate message (naming the task id and `provision-run`) verbatim in the escalation's `blocked`, worker never spawned, and still no fabricated `env-blocked`. Barrier/polish provisioning routes (outside the thunk) are byte-unchanged.
  7. An inside-thunk pt throw (the criterion-3 `task.title` case) routes to `held:escalation` carrying the pt undefined-interpolation message in `blocked`, no worker dispatched — test rewritten + retitled. pt throws OUTSIDE the thunk (Provision-barrier, merge/land, gate-audit prompts) still route to `held:workflow-error`.
  8. `FILES_CHANGED_RULE` (a canonical constant with no backtick/quote-mark tokens) is concatenated into the dispatched `Implement WAR task …` prompt and mirrored VERBATIM in `agents/war-worker.md`, locked by a byte-compare test (the `FORCE_WITH_LEASE_RULE` precedent); `schemas.md` annotates `WorkerResult.files_changed` as worktree-relative; all in one commit.
  9. The Entry validation (H) blockquote in `skills/war/references/schemas.md` documents the unconditional phase-field class, locked by a proximity-window anchor scoped to the blockquote (the `Entry validation (H)` heading token followed within a bounded window by `title`, `workingBranch`, `integrationBranch`); the legacy `/entry validation/i` anchor still matches.
  10. The token `assertReportedPathsInWorktree` appears nowhere in `workflow-template.js` or any test title — locked by a negative source assertion (the deleted-fabrication-literal precedent); the helper's comment block cites THIS spec (old citation kept parenthetically as provenance).
  11. The three token sweeps (spec §4) are executed with every match handled AND each sweep's manual same-scope title/comment survey recorded as a `Survey:` block in the implementation commit message body (auditor-verifiable at the pinned `audit_sha` via `git log`), stragglers fixed as survey-derived corrections.
  12. All four release slots bumped together to the next free patch (Phase 2).
