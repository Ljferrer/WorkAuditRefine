# /war run-lifecycle robustness — provision dispatch contract, phase-scoped worktree keying + relaunch recovery, launch-args entry validation

Source spec: [2026-07-08-war-run-lifecycle-robustness-design.md](../specs/2026-07-08-war-run-lifecycle-robustness-design.md)
Issues: #582 (provision dispatch out-of-contract) · #583 (phase-blind worktree keying + no sanctioned retry) · #586 (missing launch args die per-task, not at entry).

## Commander's Intent

- **Purpose:** the run lifecycle must fail in the *right class* at its three seams — dispatch, relaunch, launch. A refused provision dispatch must never be reported as an environment gap with fabricated evidence (#582); a same-run cross-phase relaunch must never collide with a stale worktree, and an escalated task — or a dead phase — must have a *sanctioned* retry path instead of improvised "phase 4b" surgery (#583); a launch missing top-level args must die once, at entry, naming exactly what is absent (#586).
- **Method:** convert the spec's resolved tree A–I without re-litigating it. Teach the refiner's standing card the `provision` mode it already performs (never reroute the dispatch — ADR 0001), make all three provision dispatches return the existing `ENV_OUTCOME` shape with every result captured, and gate `env-blocked` on **execution evidence** (a `failedCommand` matching a dispatched step + a numeric non-zero `exitCode`; anything else throws into the existing catch → `held:workflow-error`, and the `provisionList[0]` / `'provision-run returned no result'` fabrication is deleted). Phase-scope the derived task worktree path (`p<ph.id>-<t.id>`, mirroring `taskBranch`) and the polish worktree; `_refinery` stays run-scoped by design. Add one entry-validation check at args parse enumerating the missing trio keys — widened, operator-ratified, to also catch a missing `phase.id` (the silent `pundefined-` derivation class). Ship the **recovery relaunch** as Lead playbook prose with **two entry points** — single-task retry of an escalated/env-blocked task, and whole-phase relaunch of a `held:workflow-error` / retries-exhausted `held:phase-incomplete` phase — (fresh run, new runId, same plan slug + same numeric phase id, owned-file continuity, never `resumeFromRunId`), composing existing git-first reuse (ADR 0008). Every dispatched-prompt change lands with its `agents/war-refiner.md` mirror in the same commit (both-surfaces rule); no enum, no `land-decision.mjs`, no hook, and **no `provision-worktrees.sh`** change anywhere (spec E/F are keep-decisions).
- **End state:**
  1. **Entry validation** (`workflow-template.test.mjs`): tasks lacking explicit `branch`/`worktree` + (a) no trio → `held:workflow-error`, message lists all three keys; (b) only `runId` missing → message lists exactly `runId`; (c) trio absent but every task carries explicit `branch`+`worktree` → no throw, run proceeds; (d) trio present but `phase.id` absent → `held:workflow-error`, message names `phase.id` **distinctly** from the trio-keys class. In (a)/(b)/(d) **zero agents dispatched**; the empty-task phase still returns `held:nothing-merged` untouched.
  2. **Phase-scoped keying:** derived `t.worktree` = `<worktreeRoot>/<runId>/p<ph.id>-<t.id>`; the same taskId under two phase ids yields two distinct paths (delete-the-feature check: the assertion fails on the old phase-blind derivation); explicit `t.worktree` still wins; the polish worktree path carries `p<ph.id>-polish`.
  3. **Evidence gate:** an `ok:false` provision-run result whose `failedCommand` trim-matches a dispatched `run.provision` step (array membership, never substring) and whose `exitCode` is a number **≠ 0** → today's soft `env-blocked` escalation byte-preserved (reason `env-blocked`, worker never spawned, worktree kept, siblings proceed, 0 fix rounds); a missing result, a refusal, a foreign `failedCommand`, **or an incoherent `exitCode: 0`** → `held:workflow-error` whose message names the task and `provision-run`; the literal `'provision-run returned no result'` appears nowhere in the template.
  4. **Barrier/polish capture:** both dispatches pass `ENV_OUTCOME` (not `MERGE_RESULT`) as `schema` and both results are bound; barrier `ok:false` (with or without evidence — no topology ⇒ nothing can run) → `held:workflow-error` carrying the `stderrTail`; polish `ok:false` → sweep skipped, queue drained to `follow-up`, the phase still lands (never a hold — the invalid-roster arm's template).
  5. **Both-surfaces drift guard:** a token-anchored, sentence-case-tolerant test proves `agents/war-refiner.md` names the `provision` mode, the three dispatch labels' duties (`provision:phase-<id>`, `provision-run:<taskId>`, `polish-worktree:phase-<id>`), and the env-outcome return fields wherever the dispatched prompts do; that its **frontmatter `description`** (the agent-catalog blurb) names the provision mode / env-outcome return alongside the MergeResult; and that the standing card's submodule provisioning path matches the template's `p<phase>-<task>` derivation shape.
  6. **Doc presence (grep-able):** `skills/war/SKILL.md`'s launch step names `planSlug`/`runId`/`worktreeRoot`; a recovery-relaunch heading exists whose body names **both entry points** (single-task retry; whole-phase relaunch for `held:workflow-error` and retries-exhausted `held:phase-incomplete`), owned-file continuity, "never `resumeFromRunId`", and the keep-commits-as-WIP directive; the `held:workflow-error` prose names the provision-refusal case; the `env-blocked` bullet names the evidence gate; `schemas.md`'s provisioning-args rows carry the `p<phase>-<task>` shape; the swept doc surfaces carry `p<N>-polish` with **no stale `_polish` token** (ADR 0012 excluded as historical); `CONTEXT.md` defines the three new terms; a new ADR records the run-lifecycle contract, the polish-path rename, and the run-scoped-vs-phase-scoped naming convention.
  7. All four release slots carry the same new version; full gate green at the landing tip.

## Notes / conscious deviations

- **External ordering — already satisfied.** The spec ordered this plan after `docs/plans/2026-07-07-target-repo-agnostic-execution.md` and `docs/plans/2026-07-07-test-floor-pattern-threading.md`; **both landed on master before this plan was authored** (PR #602, v0.14.10). Every task above is grounded against that post-campaign tip (e.g. the barrier prompt already carries the ADR-0008 base-derivation prose; the merge prompts carry `testPatternArg`) — no rebase contingency is needed, and the roadmap needs no ordering edge for them.
- **`record-as-owned` does not exist** (operator-ratified substitution) — spec G names it, but `provision-worktrees.sh`'s dispatcher has no such subcommand (the phrase appears only inside a `teardown-task` die message as a concept). The playbook uses the real mechanism: `args.ownedFile` → `ensure-integration --owned-file` (the only ownership flag the template threads), with append-the-ref-to-the-owned-file as the manual equivalent. Same semantics, real surface.
- **Entry validation widened beyond the issues' ask (operator-ratified, decision 4):** #586 asked only for the trio; the same entry throw now also catches a missing `phase.id` — killing the silent `pundefined-` derivation class the trio check alone would leave open. Grounding note: the Workflow receives **one** `phase` object per run (`args.phase`), so "phases missing `id`" reduces to validating `args.phase.id`; the message names `phase.id` as its own class, distinct from the missing-trio-keys list.
- **Entry-validation placement:** the spec says "immediately after the `A` destructure … inside the existing `try{}`"; the destructure sits *above* the `try{}`, so the check lands at the top of the `try{}` body — first statement before the derivation loop — preserving both halves of the spec's requirement (early + catch-routed).
- **Recovery playbook covers whole-phase relaunch (operator-ratified, decision 2):** beyond spec G's single-task retry, the same subsection documents the whole-phase relaunch for `held:workflow-error` and retries-exhausted `held:phase-incomplete` — every hold the new hard-stop routing can produce has a documented exit. One subsection, two entry points, shared mechanics (fresh run, owned-file continuity, never `resumeFromRunId`).
- **Retried branches keep their commits (operator-ratified, decision 3):** a relaunched task branch may carry audit-rejected commits; the playbook directs fixing forward per the recorded findings (seeded into the retry `planSlice`), never a history rewrite — consistent with never-force / never-reset on shared refs.
- **Intra-phase drift window (accepted):** Task 1 (code routing) and Task 2 (SKILL.md class prose) are same-wave siblings merging serially within one phase and landing in one PR — between the two serial merges the tree briefly carries new routing with old prose (or vice versa). Accepted: no cross-task `deps` (the tasks are file-disjoint and independently green); the drift window never escapes the phase.
