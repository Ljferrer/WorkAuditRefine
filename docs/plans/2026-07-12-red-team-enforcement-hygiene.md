# Red-team enforcement hygiene — envGap typed-flag gate demotion, CONTRACTS "discarded" reword, die() default alignment

Source spec: docs/specs/2026-07-12-red-team-enforcement-hygiene-design.md
Issues addressed: #807, #808, #812

## AI-Commander's Intent

- **Purpose:** Three /red-team probe-integrity guarantees are weaker in code than in prose. The env-gap→warn rule (#807) exists only as prompt text, and even the *obedient* path misfires: a severity-less env-gap note on a non-pass probe is force-promoted to `needsDecision:true` in `classify()` (`skills/red-team/assets/red-team-gate.mjs`) and always blocks — a broken environment gets mis-scored as a broken plan. The CONTRACTS header in `workflow-scaffold.js` (#808) claims a pass-probe Critical/Major is "discarded as a non-defect" when it is only filtered from `blockers` and stays in `allFindings()`. And `assert-no-repo-escape.sh` (#812) has a `die()` helper whose default exit substitution is the escape code (1), not the conservative infra code (2) its own header contract demands — a future `die "msg"` omitting the code silently reports an infra failure as an escape.
- **Method:** The spec resolved **build, not ratify** for #807 — follow it. Enforcement lands **gate-side, typed-flag only** (the ratified `deliverableAbsence` precedent, ADR 0032): a finding-level `envGap: true` flag in the `FINDINGS` schema, stamped per the extended `provisionDirective`, demoted to Minor in `classify()` **before** the `KNOWN_SEVERITIES` branch so the severity-less obedient-path trap can never reach the `needsDecision` force-promotion. The gate stays pure (no NLP on `reality`/`evidence`); the scaffold stays runs-no-shell (the B3 deterministic provision runner stays deferred); the demotion only ever *reduces* blocking — never drops a finding, never clears `needsDecision`. `dedupe()`'s malformed-finding fallback key gains the `probe` component so two probes' identical severity-less env-gap notes cannot silently collapse to one (the never-silently-dropped guarantee holds through dedupe, not just classify). Both-surfaces rule: the dispatched `provisionDirective` string and the standing `lenses.md` "Provision step (executed probes)" bullet change in the same commit, locked by a **region-scoped** presence-pair test (the ff-topology ±window pattern — token presence alone rots). Gate demotion, schema, directive, and lenses ride ONE task so no integration tip ever carries the stamp-the-flag instruction without the demotion that honors it. #808 is a comment-only reword with a standing positive lock on the replacement wording; #812 is a one-character default flip plus a source-level default lock AND a standing negative call-site lock (every `die` call site must carry an explicit exit code — the one-time manual sweep becomes a permanent test). The tracking lesson is rewritten from "no enforcement point exists" to "enforced at `classify()`, residual = flag-setting is agent-side" — frontmatter `description`, `metadata.keywords`, and provenance move with the body.
- **End state:**
  1. `node --test skills/red-team/assets/red-team-gate.test.mjs` passes with new envGap cases mirroring the D3 `deliverableAbsence` suite (its fixture-factory shape): (a) a Critical `envGap: true` finding is absent from `blockers` regardless of probe status; (b) it surfaces in `minors` (never silently dropped); (c) a severity-less `envGap: true` finding on a non-pass probe lands in `minors`, not `needsDecision` — the branch-ordering anchor (envGap check precedes `KNOWN_SEVERITIES`); (d) controls: the same finding without the flag blocks exactly as before, `envGap: false` is untagged, and a truthy non-boolean (`envGap: 'true'`) is untagged — only `=== true` demotes; (e) a finding carrying BOTH `deliverableAbsence: true` and `envGap: true` lands in `minors` (one Minor; the `deliverableAbsence` branch wins by order — pinned so the branch order is load-bearing in both directions); (f) a finding with `envGap: true` AND agent-set `needsDecision: true` STILL lands in the `needsDecision` bucket and blocks — by design (demotion never clears user-owned ambiguity), pinned with a comment so no future pass "fixes" it; (g) two severity-less, claim-less env-gap notes with identical file/line/summary from two DIFFERENT probes both survive `dedupe()` and surface as two minors (the fallback-key `probe` component), while the same two notes from ONE probe still collapse to one (genuine dupe, #311 intent preserved).
  2. `node --test skills/red-team/assets/workflow-scaffold.test.mjs` passes with a **region-scoped** presence-pair lock (the ff-topology ±320-char-window pattern already in this test file) proving each `envGap` surface carries its load-bearing clauses, not just the token: the `FINDINGS` schema item properties carry `envGap: { type: 'boolean' }` (deepEqual, like the existing `deliverableAbsence` schema anchor); the `provisionDirective` failure clause's envGap region carries `envGap: true` and `warn`; the `lenses.md` "Provision step (executed probes)" bullet's envGap region carries `envGap: true` and, mid-sentence case-insensitively, `never` with `red/fail` (or `Critical/Major`).
